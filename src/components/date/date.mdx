- If the date is likely to be close to today (e.g. an invoice due date), then a datepicker may be useful. If the date is likely to be far in the past (e.g. a date of birth), then it may be better to use separate inputs for day, month, and year or use a NumeralDate component.
- Users can type a date directly or open the calendar. The typical variant uses a calendar button; the legacy variant retains its icon interaction.
- Carbon handles a range of formats, like dd/mm/yyyy, dd/mm/yy, dd/mm, or dd. Configuration can be regional, and copes with space, slash, full stop, comma and colon as separators, as well as no separator at all.

## Typical variant compatibility

The `typical` variant is the default. Deprecated props remain part of the shared API for existing legacy consumers, but the following legacy presentation props are not supported by typical.

| Deprecated prop     | Typical behavior                                     | Replacement                                               |
| ------------------- | ---------------------------------------------------- | --------------------------------------------------------- |
| `fieldHelp`         | String values render as the input hint               | `inputHint`                                               |
| `labelHelp`         | String values render as the input hint               | `inputHint`                                               |
| `helpAriaLabel`     | Not supported                                        | Use an explicit label and `inputHint`                     |
| `info`              | Not supported                                        | Use `inputHint`, `warning`, or `error` as appropriate     |
| `labelSpacing`      | Not supported; typical uses token-defined spacing    | Typical layout tokens                                     |
| `labelWidth`        | Not supported; typical uses token-defined dimensions | `inputWidth` or `maxWidth` where appropriate              |
| `reverse`           | Not supported                                        | Standard typical layout                                   |
| `tooltipId`         | Not supported                                        | `inputHint`                                               |
| `tooltipPosition`   | Not supported                                        | Inline `inputHint`, `warning`, or `error` content         |
| `validationIconId`  | Not supported                                        | Inline validation message ids are generated automatically |
| `validationOnLabel` | Not supported                                        | Inline `warning` or `error` messages                      |

### Custom calendar renderers

`pickerProps.components` can replace React DayPicker renderers, including `DayButton` and `Weekday`. Carbon supplies accessible, styled renderers by default. A custom renderer takes ownership of the behavior it replaces and must forward all props supplied by React DayPicker to its rendered DOM element. This includes `className`, event handlers, disabled state, and accessibility attributes.

Custom `DayButton` renderers must retain the `rdp-day_button` class supplied through `className` so that Date Input can identify calendar days for initial focus and Tab navigation. They must also forward DayPicker's keyboard and pointer handlers and provide an accessible name for the complete date. Custom `Weekday` renderers should forward `className` and accessibility attributes and provide an appropriate localized label.

```tsx
<DateInput
  pickerProps={{
    components: {
      DayButton: ({ modifiers, ...buttonProps }) => (
        <button {...buttonProps}>
          {buttonProps.children}
          {modifiers.today && <span aria-hidden="true">Today</span>}
        </button>
      ),
      Weekday: ({ children, ...weekdayProps }) => (
        <th {...weekdayProps}>{children}</th>
      ),
    },
  }}
/>
```

### Default

The `value` and `onChange` props are required as this component must be controlled. The `onChange` will receive a event
with a target value containing a `formattedValue` and `rawValue`. The `rawValue` should be used for storing the value
in the backend and for running validations: it will either return a valid ISO string or null. The `formattedValue` should
be used to set the input value like in the example below, although the component can be initialised with an ISO formatted string.

### Locale override

The examples below illustrate how to override the locale passed into the Date component. They show German (`de-DE`), Chinese

Required locales can be imported from date-fns like so: `import { de, zhCN } from 'date-fns/locale';`

<Canvas of={DateStories.LocaleOverrideExampleImplementation} />

### Locale format override

You can also override the default date format for a specific locale by passing a `dateFormatOverride` prop to the `DateInput` component or alternatively,
you can use the `date.dateFormatOverride` translation key.
The example uses a German locale with a `yyyy-MM-dd` translation-key format. The second field also passes `dateFormatOverride="dd/MM/yyyy"`, demonstrating that the component prop takes precedence over the translation key. Users should see the same date represented as `2019-04-05` in the first field and `05/04/2019` in the second.

<Canvas of={DateStories.LocaleFormatOverrideExampleImplementation} />

### With labelInline

Use the `labelInline` prop to display the label on the same horizontal row as the input. You can adjust its appearance using the `labelWidth` and
`labelAlign` props to control the width and text alignment of the label and the `inputWidth` prop to control the width of the input.

**Note:** This is a legacy feature and will only render if the `validationRedesignOptIn` feature flag on an ancestor [CarbonProvider](../?path=/docs/carbon-provider--docs) is _false_.
